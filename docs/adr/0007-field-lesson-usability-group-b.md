# ADR-0007: 필드 레슨 노트 사용성 개선 — 자동 저장 + 홀 네비게이터 (묶음 B)

**Date**: 2026-04-26
**Status**: accepted
**Deciders**: hdopen@moacoms.com
**Related**: [ADR-0006](0006-field-lesson-accuracy-group-a.md) — 묶음 A 후속

## Context

ADR-0006으로 입력 필드가 늘어나면서 두 가지 사용성 문제가 부각:

- **데이터 유실 리스크**: 필드 레슨 노트는 라운드 중·후 분량이 많은 입력. 작성 도중 앱이 종료되거나 다른 화면으로 이탈하면 입력이 모두 사라짐
- **홀 이동 비용**: 18홀 카드를 세로로 스크롤해 찾아가야 함. 특정 홀(예: 7번홀)로 빠르게 이동하기 어렵고 라운드 중 실시간 입력에 부적합

이는 묶음 A 결과로 더 심해진 부작용이라 동시 해결이 필요.

## Decision

두 기능을 묶음 B로 한 번에 도입:

1. **로컬 드래프트 자동 저장**
   - `shared_preferences`에 폼 상태를 JSON으로 저장 (이미 의존성 보유)
   - 입력 변경 시 디바운스 500ms 후 저장
   - 폼 마운트 시 드래프트가 있으면 "이전 작성 중인 내용을 복원할까요?" 다이얼로그
   - 저장(`saveNote`) 성공 시 드래프트 삭제
   - 폼 dispose 시 강제 flush로 디바운스 누락 방지
   - 키 분기: `lesson_note_draft_new` (신규) / `lesson_note_draft_edit_{noteId}` (편집)

2. **홀 네비게이터**
   - 필드 탭 상단(코스 정보 아래, 홀 카드 위)에 가로 스크롤 가능한 칩 행
   - 각 칩: 홀 번호 + 입력된 `score_label` 색상 도트 (입력됐다는 시각적 피드백)
   - 9홀×2(`nine_double`)의 경우 "1차 1~9 · 2차 1~9"로 그룹 표시
   - 칩 탭 시 `Scrollable.ensureVisible`로 해당 홀 카드까지 스크롤. ExpansionTile은 자동 펼침까지는 하지 않음(시야 안에 들어오면 사용자가 탭해서 펼침 — 단순화)

## Alternatives Considered

### 대안 1: 드래프트를 서버(Supabase)에 직접 저장
- **Pros**: 디바이스 변경 시도 복원 가능, 다중 디바이스 일관성
- **Cons**: 네트워크 의존, 부분 저장의 의미 모호, RLS·동시성 처리 추가, 실시간 입력마다 네트워크 콜 부담
- **Why not**: 로컬 드래프트가 단순하고 빠름. 서버 저장은 "실제 저장 버튼"의 역할로 충분

### 대안 2: 드래프트 저장소를 Hive/Isar 등 구조화 로컬 DB로
- **Pros**: 다중 드래프트, 검색·인덱싱
- **Cons**: 새 의존성, 마이그레이션 부담, pre-launch에 과함
- **Why not**: 노트 1건당 JSON 한 덩어리 저장으로 충분

### 대안 3: 홀 네비게이션을 별도 IndexedStack/PageView로 변경
- **Pros**: 한 번에 한 홀만 화면 → 입력 집중
- **Cons**: 기존 SingleChildScrollView 구조 전면 재설계, 전체 라운드 흐름이 잘 보이지 않음
- **Why not**: GlobalKey + `ensureVisible`로 최소 변경. 스크롤 패러다임 유지가 자연스러움

### 대안 4: 자동 저장 없이 "저장하지 않고 나가시겠어요?" 경고만
- **Pros**: 구현 단순
- **Cons**: 앱 강제 종료·OS 메모리 회수 시 무력. 라운드 중 사용 시나리오에 부적합
- **Why not**: 실제 유실 시나리오를 막지 못함

## Consequences

### Positive
- 라운드 중 앱 종료·페이지 이탈에도 입력 유지 → 실사용 신뢰도 ↑
- 18홀에서 특정 홀로 1~2초 내 이동 가능 → 라운드 중 실시간 입력 가능
- `shared_preferences`만 사용해 새 의존성 없음

### Negative
- 드래프트 저장 실패(디스크 가득 등) 시 사용자에게 알리지 않으면 조용한 유실 가능 — 일단은 silent fail로 시작
- 홀 칩 + 스크롤 컨트롤러 + GlobalKey 18~36개로 위젯 트리 복잡도 증가
- 키 충돌·디바운스 타이밍 등 엣지 케이스 테스트 필요 (수동 검증 위주, pre-launch 단계)

### Risks
- 드래프트 키가 학생/날짜 단위가 아닌 "신규 1슬롯"이라 학생 A 노트 작성 중 학생 B 노트 작성을 시작하면 A 드래프트가 덮어쓰임 — 묶음 C 이후 다중 슬롯 검토. 일단 단순 1슬롯으로 시작.
- 드래프트 JSON 스키마 변경 시 복원 실패 가능 → `version` 필드를 드래프트에 포함해 mismatch 시 무시
