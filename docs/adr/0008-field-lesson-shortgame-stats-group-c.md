# ADR-0008: 필드 레슨 노트 숏게임 통계 — 샌드세이브 + 업앤다운 + 드라이빙 평균 (묶음 C)

**Date**: 2026-04-26
**Status**: accepted
**Deciders**: hdopen@moacoms.com
**Related**: [ADR-0006](0006-field-lesson-accuracy-group-a.md), [ADR-0007](0007-field-lesson-usability-group-b.md) — 묶음 C는 A/B의 후속 통계 확장

## Context

묶음 A/B로 기록 정확도와 사용성을 보완했지만 평가에 가장 자주 쓰이는 핵심 골프 통계 3종(샌드세이브율, 업앤다운율, 드라이빙 평균 거리)이 빠져 있다. 레슨프로가 학생을 평가할 때 "총 타수"보다 이 셋을 더 본다는 도메인 사실이 있고, 없으면 통계 카드의 효용이 제한적.

## Decision

세 지표를 추가:

1. **샌드세이브** — 홀별 `sand_save: 'made' | 'missed' | null` 필드 (null = 시도 없음). 벙커에서의 성패를 프로가 직접 기록.
2. **업앤다운** — 홀별 `up_and_down: 'made' | 'missed' | null` 필드. 그린 주변에서 칩+1퍼트로 파를 만들었는가.
3. **드라이빙 평균 거리** — 새 필드 없이 기존 `shots[].distance_m` 중 `shot_type == 'tee' && club == 'driver'`에 한해 평균 계산. 통계 카드에 노출.

UI 추가: 홀 카드에 컴팩트한 3-state ChoiceChip 두 줄(`성공 / 실패 / 시도 없음`). 기본값은 시도 없음.
통계 카드: 점수 요약(`ScoreSummaryCard`) 또는 분석 카드(`FieldStatsCard`) 어느 쪽에 둘지 — 노출 빈도가 높은 핵심 지표라 **점수 요약 카드**에 추가해 항상 보이도록.

## Alternatives Considered

### 대안 1: 샷 데이터에서 자동 추론 (lie='bunker' 후 ≤2 strokes로 sand save 판정 등)
- **Pros**: 사용자가 별도로 표시할 필요 없음
- **Cons**: 추론 규칙이 모호 (그린사이드 벙커 vs 페어웨이 벙커 구분, 어프로치 거리 임계값 등). 샷 입력이 누락되면 자동 판정이 틀릴 수 있음
- **Why not**: 명시적 입력이 정확하고 단순. 라운드 후 프로가 직접 누르는 게 신뢰도 높음

### 대안 2: 샌드세이브와 업앤다운을 하나의 enum으로 통합
- **Pros**: 필드 1개
- **Cons**: 한 홀에서 둘 다 발생할 수 있는 케이스 표현 불가 (예: 벙커에서 빼고 다시 칩으로 업앤다운 시도)
- **Why not**: 분리 보존이 도메인적으로 명확

### 대안 3: 드라이빙 거리에 별도 `driving_distance_m` 필드 추가
- **Pros**: 명시적
- **Cons**: 묶음 A에서 이미 도입한 `shot.distance_m` + `club` 조합으로 충분히 도출 가능
- **Why not**: 중복 데이터

## Consequences

### Positive
- 레슨 평가의 핵심 3지표를 한 라운드 안에서 자동 집계 → 학생 피드백 품질 향상
- 드라이빙 평균은 새 필드 없이 기존 데이터로 계산 → 마이그레이션·UI 부담 0

### Negative
- 홀 카드 입력 항목이 또 늘어남(한 홀당 6칩 추가) → 모든 홀에 입력하면 부담. 기본값을 "시도 없음"으로 두어 미입력 시 자동 무시
- 점수 요약 카드 가로 폭이 늘어 모바일에서 아이콘 크기·간격 조정 필요할 수 있음

### Risks
- 사용자가 입력을 빠뜨리면 통계가 부정확 → 통계 표시 시 "기록된 시도 N건 기준" 같은 부속 텍스트로 오해 방지
- 드라이빙 평균은 `distance_m`이 입력된 티샷만 집계 → 미입력 라운드는 N/A
